The useful mental model for AI coding agents is not "fast junior engineer." It is "inexhaustible generator of plausible code that does not deserve trust." That sounds harsher than the usual framing, but it is closer to the operational truth. Teams get sloppy when they imagine understanding on the other side of the interface. There often isn't any. The hard part is not getting code out of the model. The hard part is refusing to mistake output for judgment.

That is why "prompting" is overrated relative to review. Generation is cheap. Evaluation is the bottleneck.

Human-written code usually carries signs of intent. Even bad code often tells you what its author thought they were preserving, what tradeoff they believed they were making, what assumption they were leaning on. AI-generated code can be neat, internally coherent, even idiomatic at the function level, while having no real anchor in the surrounding system. It reaches for the wrong abstraction. It recreates a helper that already exists three directories away. It handles the prompt and misses the product. It violates an unwritten invariant because unwritten invariants are exactly the kind of thing the model does not reliably know. The problem is not just bugs. It is fake understanding presented in the syntax of competence.

That should change the order of review.

With human code, you can often start at maintainability or style because you assume, at least provisionally, that the author was trying to solve the right problem inside the right conceptual frame. With AI code, that assumption is weak. The first question is cruder and more important: should this code exist in this form at all? Is it solving the right problem, through the right interfaces, without duplicating logic the codebase already has? AI is very good at producing solutions that are locally tidy and globally wrong. It optimizes for the requested shape of work, not for preserving the system's internal logic.

This is where reviewers get fooled. Fluency is one trap. Clean comments, polished formatting, and a complete-looking test file create the feeling that someone did the work of thinking. Maybe no one did. Scope creep is another. Ask for a narrow change and the model will often "improve" adjacent code, extract helpers, modernize patterns, or refactor structure simply because those moves statistically resemble finished software. None of that is free. Every unnecessary abstraction is another surface area you now own. And passing tests do not rescue you if the model wrote those too. Green checks are weak evidence when the same generator authored both the implementation and the conditions under which it passes.

So containment matters more than clever prompting. Ask for narrow edits. Ask for alternatives. Ask it to point to existing symbols and files instead of inventing fresh structure. Then review the result like a hostile reader. Every new abstraction should justify its existence against what is already in the codebase. Every deletion deserves suspicion because behavioral change hides there. Generated tests should be read as advocacy documents, not neutral evidence: what case did the model quietly decline to specify? What invariant appears to exist only because the implementation made it convenient?

On critical paths, mechanical diff review is not enough. You have to rerun the reasoning from requirements to code yourself. That is slower. It is also the part that matters.

The larger shift is uncomfortable but real: code review is no longer a secondary social ritual wrapped around authorship. It is becoming the central engineering skill. As generation gets cheaper, leverage moves to skepticism, system memory, and the ability to notice when something merely looks right. The teams that get the most from AI will not be the ones that produce the most code. They will be the ones that get good at killing plausible nonsense before it ships.
